Why

Ohio cannabinoids QC lives in three layers of Excel today: the Shimadzu instrument's per-injection text file, a per-run "DataValidation" workbook where the bench analyst pastes results and signs off (with secondary review by a different approver), and two matrix-segregated trending workbooks that aggregate LCS results across instruments and chart Total THC over time. The chain is bridged by manual copy-paste, and the most consequential check the system performs — that every customer injection is flanked by passing calibration verifications in instrument-acquisition order — happens only because an analyst eyeballs it.

Control charts in our world aren't decorative. They're the lab's process-control nervous system. The point is to detect localized vs systemic shifts in accuracy and precision before customers experience them. When several customers report aberrantly low Total THC in close temporal proximity, the chart system — sliced by calibration stock, column, instrument, analyst, reagent lot, date range — is what isolates the cause. A customer once reverse-engineered our analyst-firing date by control-charting the numbers we shipped her. We need to do that internally before customers do.

The Excel layer also locks us out of court-quality audit retrieval. When a regulator, court, or police request lands, we should be able to show — at the drop of a hat, on any sample, any day — who weighed it, who extracted it, who diluted it, who analyzed it, on which instrument, with which calibration, against which LCS lot. Today that's a hand-walk through workbook history.

§SOP-140 Constraints

These are regulatory constraints, not design choices. Confirm I read them right.

  • Analysis Batch (§3.1) = same day, same analyst, same calibration curve, same mobile phase lots. An analysis batch never spans two instruments; two instruments means two analysis batches by definition.
  • Sample Preparation Batch (§3.6) = single matrix, single prep procedure, single day, single analyst. Multiple prep batches CAN combine into one analysis batch (going downward).
  • Three matrices (§1.1): cannabis flower, concentrates, cannabis-infused products. The "PP" label in the trending workbooks is an Excel artifact that conflates concentrates and infused products.
  • CCV cadence (§9.8): at run start, every 20 samples, at run end.
  • CCV pass threshold (§9.8.1): ±10% (90–110%) of expected.
  • Failed CCV protocol (§9.8.1.2): remake and rerun once. If rerun passes, the run may continue. The instrument must be recalibrated AND all samples before and after the failing CCV must be re-acquired. A passing rerun does NOT validate the failure-window samples.
  • LCS (§9.11.1): ±3σ of mean for Total THC. If fail, associated test sample results are NOT reportable until the analyst confirms source of error didn't impact them.
  • LCS recovery path (§9.11.2): re-prep and re-analyze the LCS extract with 3 random test samples from the same batch. If reprepped LCS passes AND Total THC of the 3 reanalyzed samples is within ±10% of original, original batch results may be reported.
  • Control chart (§9.15): monitored per analyte for the LCS. Limits established continuously and updated annually.
  • Replicate reporting (§10.3.8): median of included replicates is the reported result. Exclusion requires a structured reason.

§Decisions Already Made

These are committed unless you push back. Each is a tradeoff that has been resolved.

  • Bracketing rule: the immediate structural neighbors in Acquired time (CCV before, CCV after) define the bracket — pass/fail evaluated AFTER the neighbors are identified, never AS PART of the search. Both must pass for the bracket to hold. A failed CCV is an absolute wall: cannot leap over it to find a passing CCV beyond. Last sample with no following CCV is blocked until a closing CCV is added.
  • Analysis Batch identity: 1:1 with one DataValidation workbook = one instrument run = one analyst = one calibration curve = one day. The Run Validation page is keyed on the analysis batch.
  • Prep Batch: separate concept. One prep batch can yield results across multiple analysis batches (cross-instrument splits). Sample carries both PrepBatchId and AnalysisBatchId.
  • CCV pass threshold: ±10% per SOP §9.8.1.
  • Three matrices: flower / concentrates / infused products replaces Flwr/PP throughout. The existing MatchesTab + InfusedOverrideDetector cascade in CS does this classification today; we reuse it (don't reinvent).
  • The "b" CCV/CCB suffix: empirically confirmed in the May-1 UHPLC05 run that both CCV2 and CCV2b passed (every analyte 97–103%, near-identical), with the CCV2b/CCB2b/LCB2/LCS2 cluster acquired within 32 minutes — so the "b" pair is most likely an LCS-bracketing CCV/CCB injected to confirm the system before consuming the LCS standard, NOT a rerun of a failed primary. We treat every passing CCV as a candidate bracket bound regardless of suffix; analyst intent is captured as a structured tag at run validation. Pending analyst confirmation that this matches their workflow.
  • Override-with-secondary-review: a QC failure can be overridden, but the override is structured — reason required, logged with user/time/reason, NOT effective until a second unique approver signs off. Override surfaces and resolves at the Run Validation page; Sample Queue stays clean.
  • Trending population: default all-time rolling (matches Excel + SOP §9.15.2). User-toggleable last-N (60/120/200) and per-lot-reset views as alternative viewing modes.
  • PP ±5% envelope: in-house tightening. Becomes a method-configurable second rule.
  • Court-quality audit retrieval: every analysis batch's full degree-of-freedom chain (four prep-step analysts, instrument, calibration, mobile-phase lot, balance, pipette, repeater, LCS lot, ISTD lot) persisted with the analysis-batch row and surfaced in a single audit view. One-click export.
  • Release decisions are immutable snapshots, not live recomputations. Run Validation sign-off materializes a frozen decision package: QC values observed, bounds in force, bracketing assignments, computed failure windows, override state, reviewers, sign-off identity. Subsequent parser fixes, limit revisions, or matrix reclassifications do NOT silently rewrite the prior decision; they create a new version. Without this, the same historical release would produce a different answer over time as the underlying QC facts churn.

1 ·Foundational data model

Today's QC row holds one instrument-file linkage and one acquisition position — fine for a single CCV per slot but ambiguous when more than one CCV/CCB injection lands in a slot's chronological neighborhood (the "b" case). Drop in an injection-grain entity below the existing control row so each .txt file becomes its own audit-preserved record, with the control row reframed as a slot definition the injections fulfill.

In the same chunk, capture every independent variable the workbooks track — four prep-step analysts (weighed / extracted / diluted / analyzed), instrument, balance, pipette, repeater, LCS lot, internal-standard lot, calibration method file. Court-quality audit means aggressive capture: every degree of freedom gets a structured field or explicit-null. Add a Prep Batch concept on Sample so cross-instrument prep splits are queryable.

Fix two existing parser shortfalls: the current control-attach logic is shaped for terpenes naming and silently misses cannabinoid tokens (CCV1, LCS2_<lot>, Blank); the limit-evaluation path ignores the metadata that distinguishes "limits that applied at observation time" from "current limits."

2 ·Bracketing wired through ingest

The instrument file's Acquired timestamp is the gold-standard sequence anchor — the trailing number in the filename is a per-sample replicate counter and not reliable. On every file ingest, order all of an analysis batch's files by Acquired. For every customer injection, find its immediate structural neighbors — the chronologically-adjacent CCV before and the chronologically-adjacent CCV after, regardless of pass/fail status. THEN evaluate pass/fail on those neighbors.

Passed == true cannot be in the neighbor-search predicate. A failed CCV is an absolute wall, and skipping past it to find a passing CCV beyond is exactly how illegal releases happen. The release predicate is "immediate-preceding-CCV passed AND immediate-following-CCV passed." This algorithmic detail gets an explicit unit-test fixture: CCV1(pass) → Sample → CCV2(fail) → CCV2b(pass) must NOT release the sample.

Replace today's silent skip on ambiguous matches with a hard fail that surfaces a red badge to the analyst and blocks sign-off until resolved.

3 ·Release gate, fail-closed at the Run Validation page

A customer sample is releasable only if the bracketing predicate holds for the analyte being released. The block resolves at the Run Validation page (the QCReview-tab replacement) — Sample Queue downstream of run validation stays clean.

Override workflow is keyed on the blocking finding, not on a single QC row, because for §9.8.1.2 the override scope is often "the failure window before AND after this CCV," covering many samples and analytes at once. Each override is a single durable record progressing through Proposed → Approved (or Rejected or Withdrawn), backed by a thin action log. State machine is strict: Proposed keeps blocks in force; Approved makes the override effective for the gate; Rejected reverts to blocked with no auto-escalation (escalation = new human action); Withdrawn cancels a pending proposal.

The secondary reviewer must be a different user than the proposer (DB constraint). At proposal time the system snapshots the full evidence set the reviewer will see — observed QC values, applicable bounds, bracketing computation, affected sample/analyte set — into the override record so live data churn doesn't change what the reviewer is reviewing.

4 ·Run validation page replacing the DataValidation workbook

The bench workbook's QCReview sign-off is the durability boundary. That gate gets its own URL, keyed on the analysis batch (= one instrument run = one DataValidation workbook = one sign-off scope, all the same per SOP §3.1). The page shows the per-injection grid in Acquired order with bracket assignments, calibration verification per analyte, multi-injection replicate aggregation per customer sample (median per SOP §10.3.8.1), the override-with-reason interface, and the secondary-review approval surface. Without sign-off, no customer sample acquired in this batch can release.

Drop QCReview-tab fields that are system-derivable (instrument, calibration file, analyst, dates, weights, volumes, dilution, lot numbers — all already in the schema). Only solicit truly analyst-input data: visual peak-shape pass, retention-time check pass, integration review pass, override reasons, sign-off identity.

Sign-off materializes the immutable Decision Package: QC observations, bounds in force, bracketing assignments, computed failure windows, override state, secondary reviewer identity, sign-off user/timestamp, the resulting per-sample-per-analyte release dispositions. Subsequent parser improvements / limit revisions / matrix reclassifications create a NEW decision version pointing back at its predecessor; the prior version is preserved verbatim.

5 ·Matrix-segregated trending dashboard

A new QC area page with a flower / concentrates / infused-products toggle (three tabs, not two), an analyte selector, all-time mean and ±2σ / ±3σ envelopes by default, and a separate visual treatment for the hard 90–110% CCV tolerance band — it's a regulatory rule, not a derived statistic. The chart should let a chemist see at a glance "this CCV passed the hard rule but is at +2.3σ on the trend."

Day-one stratification: filter chips for instrument, analyst (per role), LCS lot, balance, pipette, repeater. A color-by mode for visualizing one stratification dimension at a time — picking "color by analyst" reveals the analyst-fingerprint pattern. User-toggleable trend population (all-time / last-60 / last-120 / last-200 / per-lot-reset) for stress-testing stability. Click any point to drill into the underlying control row + provenance.

6 ·Backfill from existing workbooks

A one-time import that parses both trending workbooks plus their upstream DataValidation specimens, materializes synthetic historical control rows with full degree-of-freedom provenance, replaces the placeholder LCS bounds with real mean ± 3σ per (matrix × analyte), and adds the in-house ±5% envelope as a second rule.

The three-matrix classifier reuses the existing CS cascade — MatchesTab + InfusedOverrideDetector already classify cannabinoids samples into Plant / Concentrates / Infused via a runtime-tunable cascade (Tier-0 ItemBrand keyword override via Azure App Settings; Tier-1 hard-coded category overrides; Tier-2 ItemCategory keyword fallthrough). The PP-row backfill walks each historical LCS row's prep-batch linkage to its customer samples' items, runs MatchesTab on those items, and assigns the matrix only if all items resolve to one tab. Mixed → log MISMATCH and skip (analog to K17 SampleTypeClassifier semantics — never silently overwrite, never guess on conflict). Mixed-matrix or no-linkage rows land as UnknownHistoricalPP, excluded from concentrate/infused trending, queued for analyst review.

Chemist confirms the imported population is the right reference set before placeholder bounds get retired.
