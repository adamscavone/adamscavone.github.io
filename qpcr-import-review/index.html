Context — what this review found

What was built: a feature to import GENE-UP qPCR instrument files straight into the LIMS — parse the export, auto-fill each sample's Presence / Absence / Inhibited result for Salmonella and STEC, and run the SOP-149 quality-control checks. A three-way independent review (one internal pass plus two external AI reviewers) found it well-built, but flagged gaps that only surface against real instrument files rather than the clean test data the demo used.

Already being handled by engineering — no science input needed: making sample-to-well matching reliable when the instrument truncates sample IDs; confirming the three channel files (HEX / ROX / CY5) all came from the same run; checking that every sample in a batch actually received a result before the import is accepted; an access-control tightening; and a database-migration cleanup. These have clear technical fixes and are being scheduled separately.

What needs you: three questions are scientific / SOP calls, not engineering ones — they belong to the Director of Science and the Senior Microbiologist. They gate the single biggest finding: the import tool currently treats SOP-149's quality-control cycle thresholds as advisory, because the real instrument data falls outside them. Until these three are answered, the tool is making a QC judgment the SOP has not authorized.

Decision 1 of 3 · highest-stakes

Is SOP-149's amplification-control window (cycles 22–27) still correct?

SOP-149 (§13.4.4, §14.1) says the internal amplification control — the HEX channel that confirms the PCR reaction worked — should cross threshold at roughly cycle 22–27. In the actual GENE-UP export files from the lab, every well crosses at about 28–29 — including the laboratory reagent blank.

Read strictly, the SOP would mark every sample on that plate inhibited and fail the whole batch. The new import tool currently lets those batches pass by treating 22–27 as advisory rather than a hard limit. That can't stay an unspoken software default — which is authoritative, the SOP number or the instrument's real behavior?

Decision 2 of 3

Should the LIMS enforce SOP-149's numeric cycle thresholds, or trust the instrument's own call?

When a result is evaluated, the software can either independently check each cycle value against SOP-149's numeric windows (control 22–27; positive controls 5–39), or accept the GENE-UP software's own “Presumed Presence / Presumed Absence” verdict. Today's build does the latter.

This is partly downstream of Decision 1: if the SOP window is revised, enforcing it in the LIMS becomes straightforward.

Decision 3 of 3

Should the failed-QC reporting exception live inside the LIMS?

SOP-149 §14.3.3 allows a failed-QC result to still be reported when there is an assignable cause and the Scientific Director (or QA) approves it. The current import tool has no such path — if QC fails for an organism, its results are simply not imported, and the analyst falls back to the manual process with the approval handled outside the system.

Should that approval be captured inside the LIMS instead?
