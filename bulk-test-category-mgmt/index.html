Decision 1 of 6 · UX shape

How should the new button behave when clicked?

Existing DPM Early Start uses a page-level overlay: the overlay sits on top of the page, the sidebar switches to multi-select, tab switching is locked while it’s active, click Apply or Cancel to dismiss. The codebase also has a polished modal pattern (TelerikWindow, used by Sample Queue and the Resend modal).

Codex argued the new button should match Early Start — zero state-machine churn, two short-lived correction modes living in the same shell. Gemini argued for moving to a modal for codebase consistency, and migrating DPM Early Start to match.

Decision 2 of 6 · persistence

How does the swap actually happen in the database?

A sample’s test category is determined by which parent panel its assay rows point at. DPM and DPM-STEC/Sal share most of the underlying chemistry assays; DPM-STEC/Sal adds STEC + Salmonella analytes on top. So the swap has two flavors:

  1. Update-in-place: change the parent reference on existing assay rows, insert only the new STEC + Salmonella rows. Per-row analyst notes, rush flags, and manual deadline overrides on the chemistry rows are preserved.
  2. Delete-and-recreate: remove all DPM-derived assay rows for the selected sample, recreate the full DPM-STEC/Sal graph from the panel definition. Per-row analyst data is lost unless explicitly carried over.

Codex flagged the audit-trail implication: at receiving time, samples typically don’t have analyst notes yet, so the data-loss risk of delete-and-recreate is low; but the cleaner semantic is delete-and-recreate (the “test category” is conceptually the whole bundle). Gemini argued for update-in-place to be safe with future post-receive corrections.

Decision 3 of 6 · scope

Bulk TAT migration to a modal — in this round or not?

You asked me to consider moving the existing Bulk TAT tab into a modal too. Codex argued no: Bulk TAT is a multi-step workflow (base deadlines + rush override rows + applied-changes summary), not a one-shot form, and modals are the wrong shape for that shape of work. Gemini argued yes: three peer correction surfaces should look the same.

My recommendation: defer it entirely for this PR. Ship the new button on the existing overlay pattern; revisit Bulk TAT as its own focused plan only if you start seeing friction with the tab.

Decision 4 of 6 · rollout ordering

When does the DPM-STEC/Sal seed land?

The new “Dispensary Plant Material - STEC/Sal” parent panel + STEC/Salmonella child panel + the STEC + Salmonella analyte attachments do not exist in current production seed. There’s an archive script from Iteration 98 that was written for the pre-rename schema; it can’t be used as-is and needs reverse-engineering against the current entity model.

Both peers agreed the seed must run before the app deploys, otherwise the new button is a dead path. The question is packaging.
