The visible problem. The Sample Type dropdown on Assay Management's Limits tab shows duplicates: three "Capsule" entries, four "Concentrate", three "Shake/Trim", and similar collisions across about a dozen names. Visually it looks sloppy — three identical-looking choices, no way to tell which to pick. Today's PR (WI 2678) wasn't going to expose this, but it surfaces once the new admin Limits tab ships.

What the duplicates actually are. Each "Capsule" row backs a different Item Category — one for "Bulk Edible," one for "Capsule for Oral Administration," one for "Bulk Capsule." Same for "Concentrate" (Bulk vs Recalled vs Bulk Final Form vs Metered Oil) and the others. They aren't true duplicates — they're rows whose Name field happens to collide because Name was written too generically. The Item Category back-reference is what makes them distinct.

The direction is already chosen: rename, don't merge. Each Sample Type gets renamed to match its Item Category (e.g., the row currently called "Capsule" that backs "Capsule for Oral Administration" becomes "Capsule for Oral Administration"). No rows are deleted; no customer samples get re-pointed. In production, no customer sample currently references this field anyway — that classification path is dev-only via the daily backfill skill, so the cleanup is a label change with no live impact. The reputational concern is the only real driver.

What the page is for. Three decisions about how to execute the rename — how aggressively to clean adjacent infrastructure, how to handle the pre-roll edge case, and whether to fix some long-standing rot the audit happened to surface.

Decision 1 of 3 · biggest scope choice

How wide does the cleanup go?

The rename touches more than just the Sample Type list. The auto-classification rules (the engine that picks a Sample Type from a Metrc Item Brand) reference Sample Types by their name, not by an internal ID. So do a handful of code paths and a couple of seed scripts. After the rename, anything still referencing the old name silently stops working.

Decision 2 of 3 · pre-roll edge case

What do we call pre-roll Sample Types?

The rename rule ("Sample Type name = Item Category name") works cleanly for most groups. Pre-roll is the awkward case. The Item Category for OH pre-rolls is the regulatory phrase "Bulk Raw Single Serving Units" and "Bulk Infused Single Serving Units." The current Sample Type names are the much shorter and more recognizable "Preroll - Raw" and "Preroll - Infused."

The regulatory pre-roll classification logic also references those short names directly in code. If we rename the Sample Type, we either accept the awkward regulatory label everywhere, or we update the code too.

Show the technical detail

OH SampleType id 4751 (Preroll - Raw) backs ItemCategory id 66 (Bulk Raw Single Serving Units) with 17 samples; id 4753 (Preroll - Infused) backs IC 67 (Bulk Infused Single Serving Units) with 5 samples. SampleTypeClassifier.cs:43,45 Tier-0 returns these strings hardcoded for OH samples on panel ids 27/28. Carve-out preserves both name and hardcode. Strict rename requires updating the hardcoded strings in the classifier. The Item-Category-rename option avoids touching code but changes a regulatory-phrase label that's referenced more broadly.

Decision 3 of 3 · adjacent rot

Fix 15 long-standing broken classification rules in this pass?

The audit surfaced something unrelated to the rename: 15 auto-classification rules already today reference Sample Type names that don't exist in the table. Items affected include Budder, Fudge, Honey, Hot Sauce, Marshmallow Fluff, Peanut Butter, Powdered Candy, Syrup, Wax, Water Soluble Tincture, and a few others.

What happens when one of these rules matches a customer product today: the classifier returns a name that resolves to nothing. The sample stays without a Sample Type. Nobody complains because the Sample Type field isn't yet driving any production behavior (see settled banner). But the rules are silently broken.

This isn't caused by our cleanup. The cleanup just surfaced it. The question is whether to fix it now or after.
