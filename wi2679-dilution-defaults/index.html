Already settled / established by research — no input needed

Plate-vs-method nomenclature. TC ≡ REC (Total Coliforms / Rapid E.coli & Coliform plate), TYM ≡ RYM (Total Yeast & Mold / Rapid Y&M plate), TAB ≡ RAC (Total Aerobic Bacteria / Rapid Aerobic Count plate), BTGN ≡ EB (Bile-Tolerant Gram-Negative / Enterobacteriaceae plate). Same analytical methods; the 3M/Neogen Petrifilm product-name codes are the OH new-micro rename of the analyte-based codes. The WI 2679 write-up's distinction between "TC defaults to 10x" and "REC defaults to 1x" is therefore internally contradictory.

SOP-101 Rev 11 §10.3.9 is the source-of-truth rule (effective 2026-02-17). Verbatim: "For samples Dispensary Plant Material samples that have a yeast and mold action limit of 50,000 CFU/g … an additional 10-fold dilution is required … Plate 1 mL of this dilution onto separate Petrifilms that are marked as '10x'." Driven by the 50,000 CFU/g YM action limit being above the RYM countable-range cap (≤150 colonies per §10.8.7). TC/REC, TAB/RAC, BTGN/EB action limits (1,000 / 100,000 / 1,000 per §10.6.1) are within the standard plate's countable range; SOP does not mandate a default 10× for them. §10.3.10's RAC 10× is a TNTC re-analysis trigger, not a routine first-pass default.

Source-commit citation in WI 2679 is misattributed. Commit 14b9eaa0 only changed QC-control dilution to always-1× (PrepStepService.cs:1313–1352). It did NOT touch the sample-side method-based dilution. The TenxMethodCodes = {"TC","TYM"} allowlist (PrepStepService.cs:177) predates 14b9eaa0.

Admin UI for dilution defaults already exists. Admin → Assay Management → (microbial assay) → SampleType Limits tab → Default Dilutions column — multi-select chip editor of dilution factors. Persists to SampleType.DefaultDilutions (CSV of factors). Every Admin role gates the page. Today the value is persisted but ignored by the runtime stamp — PrepStepService.DefaultDilutionForLbsp consults the C#-side hardcoded allowlist instead of reading this configured value.

QC-control behavior (LFB / LFM / LMB / LRB) is locked at DF = 1 regardless of SampleType per commit 14b9eaa0. Aligned with practice. No change proposed in this plan.

Decision 1 of 5 · the load-bearing call

For "Dispensary Plant Material" samples, which methods default to 10×?

SOP-101 §10.3.9 names the 10-fold dilution rule for "Dispensary Plant Material samples that have a yeast and mold action limit of 50,000 CFU/g." Read literally, that's RYM/TYM only. The bench-sheets group RAC / REC / EB / RYM under shared Weights and Plate Lots columns, with two slots each ("01" / "02") — could be either replicate plates at the same DF, or a 1× + 10× pair per method as routine lab practice.

Decision 2 of 5

Which SampleType rows count as "Dispensary Plant Material"?

The 10× rule (whichever methods are in scope from Decision 1) applies only to dispensary plant material. The code today scopes that to {Flower, Preroll - Raw, Shake/Trim, Fresh Frozen}. SOP §10.6.1 also names "Single serving units" with the same 50,000 CFU/g YM limit.

Decision 3 of 5

Code-side fix shape

You said: "I don't love hardcoded anything." The Admin UI already exposes Default Dilutions per SampleType today — persisted but ignored by the runtime stamp. The runtime consults a hardcoded allowlist in PrepStepService.DefaultDilutionForLbsp instead of reading the configured value.

Three shapes for the fix. Each fully implements whatever you picked in Decisions 1 and 2 (the difference is who edits the rule and how new methods/SampleTypes get adopted).

Decision 4 of 5

What to do about TC@10× rows already stamped in the active Kondo DB

The existing one-shot script scripts/Backfill_TcTym_10x_FlowerPreroll.sql already ran. Rows where MethodCode='TC' on {Flower, Preroll-Raw, Shake/Trim, Fresh Frozen} sit at DF=10 today.
