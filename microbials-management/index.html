Direction is locked: Microbials administration breaks out of the shared Assay Management page into a dedicated nav surface. Each Petrifilm method (TAB, TYM, TC, BTGN) and each DNA confirmation method (STEC, SAL) is first-class — its own configuration home, free to diverge over time. Settings that are genuinely shared across a family (SOPs, units, scale factor, micro extraction volume, identity) stay at the parent assay level. Gating is by lab — Ohio sees the new surface, Michigan stays on the existing micro flow until OH signs off. The new section slots under the existing "Microbials" nav group.

Peer review brought back unanimous agreement that the current shared admin page is doing too much by branching on "this assay has methods, so render the micro view." The split is overdue. Both peer agents also flagged that the chemistry-vs-micro split should key off the assay's type, not "does it happen to have methods" — that's already part of the rebuild and isn't being decided here.

One non-obvious finding from peer review: the admin UI today writes per-method defaults to a shared sample-type field, but the actual prep runtime now reads from a per-(assay × method × sample type) table that no admin page yet writes to. Admins edit one thing; the prep flow consults another. The new Microbials Management page needs to close that gap — relabeling alone would mask it. Decisions 4 and 5 below are about how and when that gap closes.

Decision 3 of 8 · your call, no recommendation

Quality Controls — per-method or panel-wide?

Today the Quality Controls editor is rendered inside each method tab on the admin page, with the header "Panel Quality Control Definitions." But the data behind it is loaded by panel (assay), not by method — the same rows show under every method tab. The UI implies per-method; the data is panel-wide.

This is the worst-of-both shape, and both peers flagged it. It needs to pick a side. This is genuinely a lab-process question, not a code-design question — so no recommendation. Trade-offs:

If QC really is per-method in the lab: the data model needs to grow (each control row gets tied to a specific method). The Quality Controls editor stays inside each method tab and finally means what it looks like. More flexibility — TAB can have a different control plan than TYM. More admin overhead — controls have to be defined per method.
If QC is panel-wide in the lab: the editor moves out of the method tabs and lives at the parent assay level (where the SOPs and identity fields are). One control plan applies to all methods on the panel. Less admin work; less flexibility per method.
Decision 4 of 8

When does the dilution write-path land?

Background: the prep runtime already reads dilution defaults from the new per-(assay × method × sample type) table when a row exists, falling back to the old sample-type-only field when it doesn't. But no admin page writes to the new table yet. The Limits tab on the current Assay Management page still writes to the old field. Until something writes to the new table, the per-method matrix stays empty and the runtime always uses the fallback.

WI 2679 (in your working tree right now) carries a punch-list item that says "rewire the Limits tab to write to the new table." That item could land on the old page as part of WI 2679, or it could land on the new Microbials Management page as part of this feature.

Decision 5 of 8

How do we seed the per-method dilution matrix?

The new per-(assay × method × sample type) table starts empty. Existing dilution defaults live in the old sample-type-only field. We need to decide what to copy over when the new write path goes live, so admins aren't staring at a blank matrix on day one.

Decision 6 of 8

Two small Petrifilm fields the system hardcodes today — promote them?

Peer review caught two values the micro analysis screen treats as constants when they're really method properties:

  1. TNTC ("too numerous to count") sentinel. Hardcoded to 999999 CFU/g in the analysis grid. When an analyst marks a plate TNTC, the system writes 999999, which always flags above the action limit. Fine in practice today, but it's a magic number sitting where a per-method "treat as >= X" value should be.
  2. Valid plate count range. Petrifilm SOPs typically only count plates in a specific range (e.g., 25–250 CFU per plate). The system doesn't enforce or display this range anywhere — analysts know it by training, but there's no admin-set field for it.

Adding these to the new admin page is a small extra lift while we're already there.

Decision 7 of 8 · needs your input

What goes on the DNA confirmation admin surface (STEC / SAL)?

STEC and SAL aren't Petrifilm — they're presence/absence DNA confirmation. The current Assay Management page treats them the same as Petrifilm methods (same Quality Controls + Limits structure), which probably isn't right for the new dedicated surface. What does the lab actually need to configure for these?

Candidate fields (informed guess, not a recommendation — pick what's real and add what's missing):

  1. Method identity (name, code, SOP reference).
  2. Per-sample-type weight ranges, like Petrifilm has today.
  3. Per-sample-type incubation parameters if they differ from Petrifilm.
  4. Quality Controls (which controls run, at what cadence).
  5. Something else specific to DNA work — confirmation result categories, kit/lot tracking, threshold cycle settings, etc.
Decision 8 of 8

Plant Pathogens family — in this feature or later?

Plant Pathogens is the third micro family. It's similar in shape to DNA confirmation but has its own analyte list and reporting conventions. Currently it shares the same Assay Management surface as everything else.
