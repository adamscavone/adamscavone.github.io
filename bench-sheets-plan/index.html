09 / Auto-checks NEW

The 14 auto-checks the panel runs

#

What the auto-validation panel grades, against existing data only. Each is auto-derivable — no analyst input needed for any of them.

Scientific QC validity

  1. LCS recovery in spec — BatchControl.ObservedResult for control type LCS vs. BatchControl.ExpectedLow/ExpectedHigh. Fail if outside.
  2. Method blank cleanliness — BatchControl rows where ControlType = MB. Fail if any target analyte exceeds the reporting limit (or ½ RL, per SOP).
  3. Matrix spike recovery — BatchControl ObservedResult for ControlType = MS / LFM vs. limits. Fail if outside.
  4. Duplicate precision (RPD) — BatchControl.RPD for paired duplicates vs. method max. Fail if exceeded.
  5. CCV bracketing complete — InstrumentRunInjection rows with CcvKind set; verify max-customer-injections-between-passing-CCVs ≤ SOP threshold (10 per SOP-140).
  6. All controls evaluated — BatchControl.Passed non-null for every control row. Null = not yet evaluated; blocks sign-off.

Audit trail / data-model integrity

  1. Every injection resolves to one owner — InstrumentRunInjection.BatchControlId XOR SampleAssayId (exactly one set, never both, never neither).
  2. Every released sample traces back to a bench sheet — for each released sample, Sample.PrepBatchId is non-null and the prep batch is Ready or Sent.
  3. Per-sample provenance complete — for every reportable SampleAssay, SampleAssayProvenance rows exist for required ProvenanceRoles (Weighed/Extracted/Diluted/Transferred/Analyzed).
  4. Per-control provenance complete — same shape for BatchControlProvenance.
  5. Lot traceability — PrepBatch has the four batch-level lot FKs populated, OR row-level Provenance overrides cover them.
  6. Equipment calibration valid at prep time — for each LabAsset referenced via Provenance, LabAsset.Expiration > PrepBatch.PrepDate.

Workflow gates (already exist; surfaced into the panel)

  1. Every disposition is resolved — no Blocked or AwaitingClose left.
  2. No pending overrides — no QcOverrideRequest in state Proposed at sign-off.

Note: we can phase rollout — start with the most critical 6–8, expand later.

10 / Solicited inputs NEW

The 5 things the analyst must attest (truly non-derivable)

#

Things no entity captures; the analyst attests:

  1. Sample physical condition / handling anomaly. Phase separation, clogging, viscosity, leakage, foaming. Pass/fail with mandatory note on fail.
  2. Reason an injection was excluded or marked non-operative. The system records the what; the analyst records the why (carryover, vial crash, autosampler miss, conditioning).
  3. Instrument conditioning acceptable. Per your §08 note — the chemist runs the instrument after conditioning. No entity captures conditioning acceptability; analyst attests.
  4. Override justification narrative. System detects an override is needed; only the analyst can write the scientific rationale.
  5. Approved-deviation rationale. Auto-validation may flag a missing expected provenance row or an off-template prep parameter. If permissible, the analyst states why.

The attestation form is empty-when-derived — if all 14 auto-checks pass and no exclusions/overrides exist, no prompts are shown.

14 / §A7 NEW NEW

§A7 — Revalidation rules after sign-off (Codex flagged as biggest risk)

#

What happens when something underlying a Validated run changes after sign-off? Examples:

  • A late injection arrives and gets ingested for the same lab batch.
  • An override gets approved or rejected after sign-off.
  • A control gets reclassified (e.g., CCV kind changed).
  • The originating bench sheet gets edited.

v1 didn’t address this. Codex called it out as the single biggest risk: a "Validated" run whose underlying evidence has since changed is an audit problem.

Three options:

  • (a) Auto-invalidate the Decision when underlying data changes. Status flips to "Pending re-sign-off." Cleanest audit; most operational friction.
  • (b) Version-bump the RunValidationDecision (the entity already has a Version field). v1 stays as-is, v2 starts as Pending, analyst signs off again. Audit trail preserved.
  • (c) Freeze a snapshot at sign-off and ignore subsequent changes for that decision version. Decision Package is immutable. Simplest mental model; loses cross-evidence reconciliation.

Recommended default: (b) version-bump. The Version field is already there; matches what most LIMS analytics workflows expect; preserves history. Confirm or pick another.

17 / §A10 NEW NEW

§A10 — Pencil-whipping mitigation on the Sign-Off tab (Gemini’s catch)

#

Granular per-step roles + multi-analyst sign-off = real audit trail, but also creates friction. If a single analyst really did all five prep steps + primary review, asking them to click five buttons each captures the same fact and tempts them to fake-attest the others to save time.

Recommended default: shortcuts on each per-step commit button.

  • "Same as previous step" — pre-fills the user from the prior committed step. Analyst confirms once.
  • "Bulk apply (me, all unsigned steps)" — single click stamps all remaining unsigned steps with the current analyst.

Both shortcuts log explicitly as bulk-attestation in the audit (so it’s clear it was one click, not five separate confirmations) — preserves traceability without faking distinct events.
