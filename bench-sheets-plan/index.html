02 / Scope

What we’re closing — the two slices

#

Two pieces of work, in order:

  1. Cannabinoid Bench-Sheet Workflow (Ohio). Replace the bench-sheet Excel workbooks with a LIMS page that captures the same data — prep batch header (matrix, equipment, analyst per step) and per-sample rows (weight, dilution factor, extraction volume) — at the moment of prep.

03 / Decision D1

D1 — Where to capture each instrument injection

#

Decision: When a raw instrument file is saved to the database, also save one row per injection into InstrumentRunInjection, linked back to that file.

  • Applies to both QC files (LCS, blanks, controls) and customer-sample files.
  • The right moment in the pipeline is after QC results are calculated and the BatchControl link is set, but before the file is flagged "processed".
  • Right after that save, the classifier (figures out which injection is which sample / standard / blank) and the rush-bracket detector run, in the same pass.
  • If anything in the chain fails, the whole save rolls back together. No half-finished rows.

Why not earlier. Earlier in the pipeline, the BatchControl link doesn’t exist yet — we’d save the injection row missing its connection to the control material, then have to come back and patch it. Why not later. Later (only inside the QC-eval routine) would skip customer-sample files entirely. The point we picked is the moment when both sides of the linkage are present.

04 / Decision D2

D2 — Where bench-sheet data lives in the database

#

Decision: Per-step analyst stamps go on PrepBatch (the batch-level record). Per-sample weights / DFs / EVs go in a new table called PrepBatchSampleEntry. Don’t touch LBSP.

Why:

  1. The Ohio workbooks capture analyst initials per step (Weighed by, Extracted by, Diluted by, Analyzed by) at the batch level — there’s no per-sample initial cell. So that’s the right grain.
  2. One prep batch can feed multiple analysis batches. If we crammed sample-level data onto LBSP, we’d duplicate it across analysis batches and have to keep the copies in sync by hand — a known failure mode.
  3. Replicates are already first-class on LBSP (each rep is its own row). PrepBatchSampleEntry follows the same convention — three reps = three entries.
  4. The Analysis tab can pick up bench-sheet data via a small change to its existing SQL view. No LBSP schema churn.

Why not "do both" / duplicate it. A duplicate-data trap we’d have to keep in sync manually is the exact pattern that bit Pearson’s micro work when fields drifted. Skip it.

05 / Decision D3

D3 — Two named surfaces on the QC summary box

#

Decision: Keep the existing "Review" button (it opens a checklist popup that’s useful and low-effort). Add a second button — "Run Validation" — that opens the new QC Run Validation page.

Two surfaces, two purposes:

  • Review (popup): SOP checklist, scoped to the method. Already exists, in daily use.
  • Run Validation (page): Per-injection detail, override workflow, frozen Decision Package, secondary-review sign-off. New.

Why not fold it into the Analysis tab inline. The chem-QC UI has been live since mid-April; analysts use it daily. The new override workflow is too much to wedge in without restructuring — and we have a UI Scope Lock rule against drive-by restructures.

Why not collapse the popup into the page. SOP checklist is per-method and form-shaped. Run validation is per-run and grid-shaped. Different mental models. One surface forced to do both is a UX smell.

09 / Slice 1

Slice 1 — How saving, status, replicates, and confirmations work

#

How saving works

Per-row commit buttons, plus autosave when you tab out of the active row. Same pattern as the microbial prep tab. Matches the Excel-era expectation that you don’t lose work if you walk away. If you refresh mid-prep, the active row’s unsaved cell restores; committed rows are durable.

Status state machine

Three values, mirrors the Excel templates’ status field:

  • In Process — created, editable.
  • Ready for Analysis — bench sheet complete, lead analyst signed off. Read-only except for the lead analyst (or supervisor).
  • Sent to Analysis — claimed by an analysis batch. Read-only.

"Ready for Analysis" → "Sent to Analysis" transitions automatically when the prep batch first gets pulled into an analysis batch.

Replicates

PrepBatchSampleEntry has a 1-based replicate index. PPPrep "Rep #" and FlwrPrep "-a/-b" both map to this — the page picks the display notation by lab/template. Three reps = three rows, all valid.

Confirmation re-runs

PrepBatchSampleEntry has a nullable "source prep batch" pointer. Set when a row is a re-run from a prior batch (the Confirmations tab pattern in the Excel templates, or a "Curaleaf Rush — PR141038 in LIMS" mid-batch insert). The Comments column carries the rush note.
