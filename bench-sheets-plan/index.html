02 / Scope

What we’re closing — the two slices

#

Two coupled slices, dependency-ordered:

  1. Cannabinoid Bench-Sheet Workflow (Ohio). Replace the 50-workbook Excel pipeline with a Razor surface that captures the canonical bench-sheet schema (PrepBatch metadata + per-row weights/DFs/EVs) at point-of-prep. Operational flex: any of weighed / extracted / diluted / analyzed can be a different analyst.
  2. Batch Results QC Visibility + Ingest Wire-up. Surface the new injection-grain in the existing chem-QC UI on /batchworkflow, wire InstrumentRunInjection writes into the file-ingest path, and connect the existing QcSummaryBlock to the new /workflow/qc-run-validation/{labBatchId} page that today’s session already shipped as a route shell.

Slice 1 is upstream because the bench-sheet workflow is the only path that produces accurate DOF (sample weight × extraction volume × dilution factor) data at point-of-prep. Backfill / from-memory entry is the failure mode the workflow is designed to prevent.

03 / Decision D1

D1 — InstrumentRunInjection write site

#

Decision: New method ResolveOrCreateInstrumentRunInjectionAsync on RawDataService, called from two places:

  • FilerController.ProcessQcControlFileAsync between the existing CreateQcRawResultsAsync (line 514) and the Processed = true mark (line 518).
  • RawDataService.ProcessRawDataFiles (the customer-sample path) at the equivalent point.

The method (a) resolves or creates the parent InstrumentRun row by (LabBatchId, InstrumentTypeId, AcquisitionDate-of-run) clustering, (b) inserts an InstrumentRunInjection keyed off the just-saved RawDataFile.Id, (c) returns the injection so the caller can decide whether to run downstream classification.

RunAssemblerService.ClassifyAsync and RushBracketDetector run synchronously at the end of the QC-eval pass (after EvaluateQcFromRawResultsAsync finishes), since they read both the new injection rows and the BCAR rows.

Why not the alternatives. Putting the write inside EvaluateQcFromRawResultsAsync (option 1c) couples it to QC-only files; putting it before CreateQcRawResultsAsync (1b) requires BatchControl resolution that hasn’t happened yet (InstrumentRunInjection.BatchControlId is the FK that needs TryStampBatchControlForTypedQcAsync at line 508 to have run first).

Idempotency. Insertion is upsert on RawDataFile.Id (one injection per file). The existing TransactionScope around lines 406+ rolls everything back together if any step fails.

04 / Decision D2

D2 — Bench-sheet data on PrepBatch + new PrepBatchSampleEntry, not LBSP

#

Decision: Use the new schema (option 2b). Keep LBSP untouched.

The Ohio bench-sheet workbooks unambiguously capture per-step (analyst, date) at the batch level — there is no per-sample initial cell. So PrepBatch.{Prepared,Weighed,Extracted,Diluted,Analyzed}ByUserId is the correct grain for analyst stamps. Per-sample rows (Sample ID, Sample Type, Weight g, DF, EV L, Comments, Seeds?) need their own home — a new PrepBatchSampleEntry entity.

Rationale, in priority order:

  1. SOP-140 §3.6 fidelity. Per-step user is a property of the prep event, not the sample.
  2. Cross-instrument prep splits. Per PrepBatch.cs:11–15, one prep batch can yield results across multiple analysis batches. Storing weight/EV/DF on LBSP would force replication across analysis batches and a denormalization invariant.
  3. Replicate convention is already on LBSP. Each replicate is its own LBSP row (unique index at EntityMaps.cs:346 includes SampleRepNumber). PrepBatchSampleEntry mirrors this — three reps = three entries. The link from entry to LBSP is established when LIMS samples flow from prep into the analysis batch.
  4. Bench-sheet → Analysis projection cost is low. The existing Analysis tab reads from LabBatchSampleView2; the view is updated to JOIN PrepBatchSampleEntry on Sample.PrepBatchId. No LBSP schema churn.

Why not “both” (2c). A denormalization invariant we have to maintain by hand is the failure mode that bit Pearson’s micro flow when fields drifted. Skip it.

05 / Decision D3

D3 — Run Validation deep-link is a SECOND command on QcSummaryBlock

#

Decision: Keep the existing OnReview → ReviewChecklistDialog modal (it’s an SOP-checklist review surface, useful, low-burden). Add a SECOND command — “Run Validation” — that navigates to /workflow/qc-run-validation/{labBatchId}. Two named surfaces:

  • Review (modal): SOP-driven checklist review, scoped to method/sub-method. Existing.
  • Run Validation (page): Injection-grain detail, override workflow, frozen Decision Package, secondary-review sign-off. New.

The “Review” name stays; the new button reads “Run Validation” (or “QC Sign-Off” — TBD, see open question §A1 below). Both render in the QcSummaryBlock group-header surface.

Why not extend the Analysis tab inline (3a). UI Scope Lock rule. The 2026-04-19 chem-QC UI ships in production; analysts use it daily; the new override workflow is too much UI to fold in without restructuring.

Why not collapse Review into the new page. SOP checklist is per-method and form-shaped; injection-grain validation is per-run and grid-shaped. Different mental models. Single surface is a UX smell.

#

Two-pane Razor:

09 / Slice 1

Slice 1 — Commit semantics, status machine, replicates, confirmations

#

Commit semantics

Explicit-button commits per row, autosave on cell-blur for the active row. Mirrors MicroPrepTab’s pattern (PrepStepService methods called on commit) and matches analyst expectations from the workbook era (Excel autosaved). Mid-prep crash recovery: the active PrepBatch has a RecoveryToken cookie so a refresh restores the unsaved cell, and committed rows are durable.

Status state machine

PrepBatchStatus enum with three values mirroring the FlwrPrep BatchID!B6 field:

  • InProcess — default at PrepBatch creation. Editable.
  • ReadyForAnalysis — bench sheet complete, lead analyst signed off. Read-only except for the lead analyst (or supervisor).
  • SentToAnalysis — claimed by an Analysis batch via LabBatchPrepBatch. Read-only.

Transition ReadyForAnalysis → SentToAnalysis is automatic on first LabBatchPrepBatch insert.

Replicate handling

PrepBatchSampleEntry has ReplicateIndex (int, 1-based). Both PPPrep “Rep #” and FlwrPrep “-a/-b” map to this — display formatter chooses the surface notation by lab/template preference. Three reps = three rows, all valid.

Confirmation re-runs

PrepBatchSampleEntry has nullable SourcePrepBatchIdInLims (FK to PrepBatch). Set when the row is a re-run from a prior batch (FlwrPrep Confirmations tab pattern, or the “Curaleaf Rush — PR141038 in LIMS” mid-batch insert from PPPrep). Comments column carries free-text rush notes.
